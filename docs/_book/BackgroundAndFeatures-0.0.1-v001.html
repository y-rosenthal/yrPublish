5  BackgroundAndFeatures-0.0.1-v001.qmd

The origins of the yrPublish system grew out of my working with the Quarto system and creating some customizations for myself. In this document I try to capture the cusomizations that I created and used across several different quaro projects that I wrote, maintain and continue to develop.

5.1 Background

I have been creating several Quarto “book” projects that I render to HTML. Most of these are to support the classes that I teach. Some of these “books” have gotten quite large. I add material pretty regularly and these books gorw in size dynamically as time goes on. I keep the source code checked into git/github.

For most of my books I have a github repo whose name ends in “Src”, e.g. myBookSrc. I have another github repo with the same base name, but that does NOT have the Src suffix, eg myBook. The Src repo contains the source code for the repo (mostly Quarto .qmd files). When I’m ready to publish the next version of the book to the Internet, I render the Src repo with Quarto. I then copy the HTMl/css/javascript and other rendered assets to my repo that does not have the Src extension.

So far, I have been hosting the rendered books in Github and serving them via github Pages. Sometimes I have more than one version of the book that I host at different github Pages sites. For example

  • rnotesSrc is a repo that contains the source code for a book I wrote about R.

    rnotes is a repo that contains just the rendered version of the book. I host access to this book via github pages at https://y-rosenthal.github.io/rnotes

    rnotesLatest is another github repo where I host the bleeding edge version of the rendered rnotesSrc book.

Of couse the rendered versions of the books can be hosted anywhere, not just github Pages. However, github Pages is convenient for me for now.

5.2 AI assisted development

I am using AI to help me develop the yrPublish product. Therefore I have structured this repo in a way that I think best aids AI in helping me.

5.2.1 docs folder

All documentation for the system, including user guides, as well as development documentation such as SPEC files are located in the docs folder. The docs are generally Quarto .qmd files and the entire docs folder is rendered as a “plain” Quarto book (i.e. not a yrPublish rendered book).

5.2.2 sampleBooks folder

While developing this repo I thought it would be helpful for the AI to have access to the actual code from some of the book repos that I have been developing. I started with just one, so as not to overwhelm the AI. This one book is located in sampleBooks/DataManagementUsingRSrc. This has the .git folder for the repo in it also so that if desired, the AI can review the git history for that repo.

5.3 For large books I currently use the Quarto “part” concept

For larger books, I use the quarto “part” concept to segment the book for easier navigation. For example these are some books that I created: https://y-rosenthal.github.io/DataManagementUsingR/ https://y-rosenthal.github.io/excelNotes/ https://y-rosenthal.github.io/rnotesLatest/ (see screen shots)

5.3.1 -v00n version numbers

On many of my files, I include a “-v00N” version number that I increment when I create a new version of the file. I started using this system before I started to use git. However, even though I started to use git I find that it is still helpful sometimes to have a version number directly in the name of a file. In that way, if I use the same file in different books, I can very easily see which version of the file I’m using. For example, I am currently up to v004 for the yrChapterNumber Bash script. Therefore, the actual name of the script that I would run would be “yrChapterNumber-v004.sh”. For Bash scripts and utilities, I generally only increment the -v00N number when I make breaking changes. So I might checking numerous non-breaking changes to yrChapterNumber-v004 (e.g. changing the comments).

For .qmd files I have a bunch of -v00N files from before I started using git. Now that I am using git, I just kept the same filenames. Sometimes, I might create a new file with a -v00N extension even though I’m using git. The motivation is often that I want more than one versin of a file to be immediately available in the repo for convenience purposes.

This may not be the best system but it works for me now. It’s a system with very loose rules. I just want you to know what the -v00N endings on many files are in my repos.

5.4 Custom chapter numbers (enabled via css and javascript at runtime)

5.4.1 motivation

I often modify the books in the middle of my teaching a course. Sometimes, I might add a chapter in the middle of the book which by default would cause the later chapter numbers to be incremented. I didn’t like that since I didn’t want to change the chapter numbers in the book in the middle of a course after students had already become familiar with the book’s layout. My thought was to label a new chapter that appears after chapter 12 for example, not as chapter 13, but rather as chapter 12b. In that way I could keep all existing chapter numbers. Unfortunately, Quarto doesn’t seem to have a standard way to have custom chapter names. Therefore I devised my own solution using custom markup and Javascript.

5.4.2 custom markup

I added a element to every first level heading in every page in the book. I then also added custom comments to the _quarto.yml at the end of every chapter line in the yml file that shows the preferred chapter number. For example, the following might be in a _quarto.yml. It shows the chapter numbers I want after # yrChapterNumber. In the following example (there is no actual book with the contents of this example), I might have had a book with 5 chapters (that didn’t have the emptyTags.qmd chapter). Suppose, I would want to add the emptyTags.qmd chapter in the middle of the semester. I would want the chapter number to be 1b and keep the rest of the chapter numbers the same.

- part: Intro to HTML
  chapters:
  - introToElements.qmd      # yrChapterNumber 1
  - emptyTags.qmd            # yrChapterNumber 1b
  - pageStructure.qmd        # yrChapterNumber 2
  - tableTags.qmd           # yrChapterNumber  3

- part: Intro to CSS
  chapters:
  - styleTag.qmd            # yrChapterNumber  4
  - basicSelectors.qmd      # yrChapterNumber  5

To support this, all .qmd files would have single top level # header at the top of the file. The text of the # header in a .qmd file becomes the text displayed in the table of contents (TOC) on the left of the Quarto book website. To support custom chapter numbers I added a element in the single top level # heading in all .qmd files in the Quarto book. I also created a Bash script, yrChapterNumber.sh, that gets the preferred chapter numbers from the _quarto.yml and populates the elements with the correct chapter numbers. I would run this bash script prior to rendering the book. So for exmple after running the yrChapterNumber.sh script I might have the following:

The first line of the “introToElements.qmd” file might ha: # 1. Introduction to HTML element

The first line of the “emptyTags.qmd” might be: # 1b. Empty tableTags

The first line of the pageStructure.qmd file might be: # 2. Structure of a valid HTML page

etc.

5.6 Cursor AI answers (from Cursor’s AI)

5.6.1 Why did I use Rscript in post-render but not in pre-render?

Cursor AI answer: Quarto’s project pre-render / post-render scripts are supported for multiple languages (including R) via Quarto’s project-script mechanism. So the “correctness” generally does not depend on whether the entry is written as bare somefile.R or as an explicit Rscript somefile.R invocation. It’s most likely historical/style/convenience: pre-render: copy_quartoYml_to_bookFolder-v001.R is already valid, and post-render: Rscript yrZipQmdFiles.R is effectively just making the interpreter explicit.

5.6.2 Is there any reference to or use of the .zip files created in post-render?

Cursor AI answer: In your described setup, the .zip files are created into _book and are listed under resources: in _quarto.yml, which means they are intended to be published as static assets. However, that does not automatically mean the existing site UI/JS references them. In most implementations of this pattern, the zips are used for “downloadable source” (or archival) even if you haven’t yet added a link/button per page. To confirm actual usage, you’d look for references to those zip filenames in templates/HTML/JS; based on the repo structure here, the published assets exist, but the linkage may be unimplemented.

5.6.3 Why include _quarto.yml in resources: and also copy _quarto.yml to _book in pre-render?

Cursor AI answer: resources: is a Quarto mechanism for shipping static files into the output directory, so including _quarto.yml there ensures it becomes available in the built output. Separately, your pre-render step copies _quarto.yml into _book so runtime client-side JS (like yrstuff-v001.js) can fetch _quarto.yml from the final site. Keeping both is likely redundant: one mechanism may have been added to guarantee a working behavior during experimentation, or to ensure timing/paths matched what the JS fetch expects. Either way, the functional goal is the same: make _quarto.yml reliably available under _book/ at runtime.

5.6.4 Why include render: - "*.qmd" and also explicitly list one older .qmd?

Cursor AI answer: For type: book, the chapters: list controls what Quarto includes in navigation, but Quarto may still skip rendering some files depending on project configuration and incremental rendering. render: "*.qmd" is a way to force rendering of all .qmd files (even if they are not in chapters). The additional explicit line for an older version (e.g. assignment-api-...-v001.qmd) is typically for “keep the old version” / “ensure it still renders” behavior: it forces the old version to be rendered even if later you switched chapters: or removed it from the main book structure.
